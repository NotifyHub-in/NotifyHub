System design

Built like a control plane, not a handoff script.

Requests enter once, get normalized, and move through policy, templates, routing, dispatch, callbacks, and observability.

Overview
Ingress Policy resolution Execution Callbacks Observability
Ingress

API + Kafka

The API validates and persists requests, then hands delivery orchestration to the worker through Kafka.

Principle The upstream service stays generic. Provider-specific behavior remains inside NotifyHub.
Resolution

Policy decides how the event behaves

Templates

Template selection can vary by channel, language, and client policy.

Preferences

User preferences gate which channels are eligible before dispatch begins.

Bindings

Bindings connect a logical event to the right channel and provider configuration.

Retry policy

Retries and fallback paths stay declarative, not hardcoded in the client.

Execution

Worker + connectors dispatch the message

The worker resolves templates, preferences, and bindings before dispatching through the relevant connector.

client request -> api -> kafka -> worker -> connector -> provider
Return path

Callbacks and replies come back through the same control plane

Provider delivery updates and inbound replies are normalized into durable events, then stored and optionally forwarded to subscribed systems.

Operations

Metrics and logs stay beside the runtime

Component What to watch
API Request rate, auth failures, validation failures
Worker Delivery latency, retries, provider failures
Connectors Provider response codes, transport errors
