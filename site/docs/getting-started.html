Onboarding

How a client uses NotifyHub.

A client registers once, sends canonical requests, and lets NotifyHub own routing, providers, callbacks, and retries.

In this guide
1. Register the client 2. Send a canonical request 3. Consume lifecycle updates 4. Apply preferences and policy Next steps
Step 1

Register the client

Create a client identity and receive an API key. The client uses that key on every request.

POST /v1/clients
{
  "tenant_id": "tenant-a",
  "client_name": "billing-service",
  "allowed_channels": ["email", "sms", "whatsapp", "push", "webhook"]
}
Step 2

Send a canonical request

Send the event name, template key, recipients, channel list, and render variables in one request. Do not include provider credentials.

POST /v1/notification-requests
Authorization: Bearer <api_key>
{
  "client_id": "client_123",
  "event_key": "login_otp",
  "template_key": "farm_login_otp_wa_v2",
  "channel": "whatsapp",
  "recipient": { "phone": "+918700491033" },
  "variables": { "otp": "123456" }
}
Step 3

Consume lifecycle updates

NotifyHub emits normalized lifecycle events back to your system so you can update your own records and dashboards.

Design rule Keep callbacks generic: treat each delivery status as a normalized event, not a provider-specific payload.
Step 4

Apply preferences and policy

Preferences

User preferences filter which channels are eligible before dispatch.

Policies

Policies decide provider selection, fallback order, template language, and retry behavior.
